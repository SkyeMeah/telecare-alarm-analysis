# Telecare Alarm Analysis

A small exploratory data analysis project looking at patterns in telecare alarm response data.

## Why

I work as a Telecare Response Officer. Day-to-day, our team responds to a steady flow of emergency alarms across the city. The patterns in those calls (when they happen, what kind they are, how long it takes us to respond) feel obvious anecdotally, but I wanted to see if I could put numbers to them and produce something that would be useful to a service planner.

Real customer data is, of course, off-limits. The data in this repo is **synthetic** generated by `generate_data.py` to mimic the shape of what we see at work, with realistic distributions but no real customers, addresses or call details.

## What's in the repo

- `generate_data.py`: produces a CSV of 5,000 synthetic alarm events
- `analysis.py`: loads the CSV and answers four questions
- `data/alarms.csv`: the generated dataset (regenerable)
- `figures/`: output charts produced by the analysis

## The questions I asked

1. What time of day produces the most alarms?
2. Are response times correlated with priority level?
3. Which alarm types take the longest to resolve?
4. Is there a weekday vs. weekend pattern in alarm volume?

## Findings (on the synthetic data)

Documented at the bottom of `analysis.py`. Even on synthetic data, two of the patterns lined up with what I'd expected from work, and one surprised me. That's the value of doing this kind of analysis on real service data: anecdotes can be wrong, and having the numbers in front of you changes the conversation.

## Running it

I'm trying to follow good Python practice from the start, which means using a **virtual environment** so this project's dependencies don't clash with anything else on my machine. If you've never done this before, the steps are below.

### 1. Create and activate a virtual environment

A virtual environment ("venv") is just an isolated copy of Python that lives inside the project folder. Anything you `pip install` while it's active only affects this project, which keeps your global Python install tidy.

**macOS / Linux:**

```bash
python3 -m venv .venv
source .venv/bin/activate
```

**Windows (PowerShell):**

```powershell
python -m venv .venv
.venv\Scripts\Activate.ps1
```

You'll know it worked when your terminal prompt shows `(.venv)` at the start of the line.

### 2. Install the dependencies

```bash
pip install -r requirements.txt
```

### 3. Run the project

```bash
python generate_data.py   # produces data/alarms.csv
python analysis.py        # produces the four charts in figures/ and prints a summary
```

### 4. When you're done

```bash
deactivate
```

This drops you back out of the venv. The next time you come back to the project, just re-run the activate command from step 1 (you don't need to reinstall the dependencies)

Charts are written to `figures/`.

## What I'd do next

If I had access to anonymised real data and the time, I'd want to look at:

- Whether response time correlates with subsequent escalation to A&E
- Whether certain device types (fall detectors vs. pendant alarms) generate disproportionately high false-positive rates
- Whether installing additional environmental sensors reduces follow-up alarm volume for the same household

These are the kinds of questions where good data and AI tools could meaningfully improve the service.
